# Release Notes

This page lists breaking changes and notable new features. Please be sure to read the release notes before upgrading from a previous version.

<Note>

The detailed changelog can be found in [TSTyche repository on GitHub](https://github.com/tstyche/tstyche/blob/main/CHANGELOG.md).

</Note>

## TSTyche 7.0  [#tstyche-7-0]

TSTyche 7.0 ships with the new `.toBeInstantiableWith()` matcher and redesigned `tsconfig` option.

### Matchers

#### `.toBeInstantiableWith()` [#to-be-instantiable-with]

Checks if a generic is instantiable with the given type arguments.

```ts twoslash
import { type _, expect } from "tstyche";

interface Matchers<R, T = unknown> {
  [key: string]: (expected: T) => R;
}

expect<Matchers<_>>().type.toBeInstantiableWith<[void]>();
expect<Matchers<_>>().type.toBeInstantiableWith<[void, string]>();

expect<Matchers<_>>().type.not.toBeInstantiableWith<[]>();
```

<Tip>

The `_` type can be used to fill in the required type arguments.

</Tip>

This matcher works with generic interfaces, type aliases, classes and functions:

```ts twoslash
import { expect } from "tstyche";

function map<T, U>(array: Array<T>, transform: (item: T) => U): Array<U> {
  return array.map(transform);
}

expect(map).type.toBeInstantiableWith<[string, number]>();
expect(map).type.not.toBeInstantiableWith<[string]>();
```

#### `.toRaiseError()`

<Warning>

**Breaking!** The `.toRaiseError()` matcher is deprecated and planned to be removed. The suggested replacement is the ability matchers (like [`.toBeCallableWith()`](../reference/expect-api.mdx#to-be-callable-with) or [`.toBeConstructableWith()`](../reference/expect-api.mdx#to-be-constructable-with)). They communicate the intent of a test more clearly and work without introducing type errors in the test code.

To learn more, see the [Expect Errors](../guides/expect-errors.mdx) page.

</Warning>

### Configuration

<Warning>

**Breaking!** Default configuration file location has changed. Now `./tstyche.json` is loaded instead of `./tstyche.config.json`.

</Warning>

#### Schema

The configuration schema now ships with the package and can be referenced locally:

```json
{
  "$schema": "./node_modules/tstyche/schemas/config.json"
}
```

Thanks to [@f15u](https://github.com/f15u) for the idea!

#### `checkSuppressedErrors` [#check-suppressed-errors]

TSTyche now checks errors suppressed by `@ts-expect-error` directives in multi-line comments.

This is useful in contexts where single-line comments are not supported. For example, in TSX files:

```jsx
function ProfileCard(props: { userName: string }) {
  return <>{props.userName}</>;
}

<>
  {/* @ts-expect-error  Type 'boolean' is not assignable to type 'string'. */}
  <ProfileCard userName={true} />
</>;
```

#### `rootPath` [#root-path]

<Warning>

**Breaking!** The `rootPath` configuration option is removed.

</Warning>

The root directory now defaults to the current directory. To use a different path, pass it with the [`--root`](../reference/command-line.mdx#root) command line option.

#### `tsconfig` [#tsconfig]

`baseline` is the new name of `ignore`.

```diff
{
- "tsconfig": "ignore"
+ "tsconfig": "baseline"
}
```

##### Inline JSON

TSConfig can be provided as an inline JSON string.

Configuration file:

```json
{
  "tsconfig": "{\"extends\":\"./tsconfig.json\",\"compilerOptions\":{\"lib\":[\"es2020\"]},\"include\":[\"**/*\"]}"
}
```

Command line:

```shell
tstyche --tsconfig '{"extends":"./tsconfig.json","compilerOptions":{"lib":["es2020"]},"include":["**/*"]}'
```
