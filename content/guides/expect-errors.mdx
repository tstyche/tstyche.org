# Expect Errors

When testing invalid code, use the ability matchers to avoid the `@ts-expect-error` trap. If the error message matters, TSTyche can also verify what `@ts-expect-error` suppresses.

## The `@ts-expect-error` Trap

Using `@ts-expect-error` directives in type tests is a popular approach to test that a certain piece of code is invalid.

In the following example, the import was lost during refactoring. The code will compile and the test will appear to pass, although the actual error is `Cannot find name 'add'`:

```ts twoslash
// @ts-expect-error Requires 2 arguments
add(2);
```

Because `@ts-expect-error` suppresses *any* errors on the following line, type tests using it cast too wide a net. This results in misleading tests. The APIs under test might change, be renamed, or be removed, yet the tests will miss regressions.

## The Ability Matchers

TSTyche ships with the ability matchers like `.toBeCallableWith()` that address the `@ts-expect-error` trap:

```ts twoslash
// @errors: 2304
import { expect } from "tstyche";

expect(add).type.not.toBeCallableWith(2);
```

The ability matchers communicate the intent of a test and can handle complex cases like generic types:

```ts twoslash
import { expect, test } from "tstyche";

type FormFieldGetter = <T, K extends keyof T>(form: T, field: K) => T[K];

const userForm = {
  name: "Alice",
  email: "alice@example.com",
};

test("FormFieldGetter", () => {
  expect<FormFieldGetter>().type.toBeCallableWith(userForm, "name");
  expect<FormFieldGetter>().type.not.toBeCallableWith(userForm, "age");
});
```

<Tip>

To make it clear why a negated assertion passes, pair it with a positive one, changing only a single element. In the example above, `"age"` is the culprit.

</Tip>

The following ability matchers are available:

- [`.toAcceptProps()`](../reference/expect-api.mdx#to-accept-props) checks if a JSX component accepts the given props
- [`.toBeApplicable`](../reference/expect-api.mdx#to-be-applicable) checks if a decorator is applicable to the given class or class member
- [`.toBeCallableWith()`](../reference/expect-api.mdx#to-be-callable-with) checks if a function is callable with the given arguments
- [`.toBeConstructableWith()`](../reference/expect-api.mdx#to-be-constructable-with) checks if a class is constructable with the given arguments
- [`.toBeInstantiableWith()`](../reference/expect-api.mdx#to-be-instantiable-with) checks if a generic is instantiable with the given type arguments
- [`.toHaveProperty()`](../reference/expect-api.mdx#to-have-property) checks if a type has the given property

<Note>

The available matchers cover most of the cases where invalid code is being tested. If you find something missing, please open an issue.

</Note>

### How They Work

The ability matchers work by erasing their syntax.

```ts twoslash
// @noErrors
expect(add).type.not.toBeCallableWith(2);
      (add)                          (2); // after erasing
```

The test file is checked for type errors, the matcher syntax is erased (as shown above) and the code is checked again. If there are new errors, the assertion is reported as failing or, when negated with `.not`, as passing.

The ability matchers are similar to `@ts-expect-error`, but are far more precise and expressive.

## When the Error Matters

For complex type interactions, `@ts-expect-error` is the most natural fit:

```ts twoslash
import { test } from "tstyche";

declare function pipe<T>(source: T, ...target: Array<(source: T) => T>): void;
declare function pick<T, K extends keyof T>(key: K): <K extends keyof T>(object: T) => Pick<T, K>;

test("pick", () => {
  pipe({ valid: true }, pick("valid"));
  // @ts-expect-error Argument of type '"invalid"' is not assignable to parameter of type '"valid"'
  pipe({ valid: true }, pick("invalid"));
});
```

`.not.toBeCallableWith()` cannot be used here, because the type error only surfaces when `pick()` is passed to `pipe()`, a broader context that the matcher cannot replicate.

In these cases, TSTyche reads the message after `@ts-expect-error` and verifies that it matches the actual error.

<Tip>

When a type is designed to produce a specific error message, `@ts-expect-error` with a message is a good way to verify it.

</Tip>

### Syntax

The following syntax can be used to adjust matching of `@ts-expect-error` messages.

To truncate parts of a long error message, use `...`:

```ts
// @ts-expect-error Argument of type ... is not assignable to parameter of type 'never'
```

To ignore a directive, append `!` to the directive:

```ts
// @ts-expect-error! This is wrong
```
